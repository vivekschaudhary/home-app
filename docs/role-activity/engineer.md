- **2026-06-23 · AAL2 marker had an absolute TTL, no sliding renewal → "forced logout every few hours"**
  - **Context:** `/fix` triage of a forced-logout report (single user). Diagnosed from the auth/session code, not the user.
  - **Pattern surfaced:** A short-lived security marker (`wlt_mfa` AAL2 token, 1h TTL) was minted ONLY at the second-factor ceremony (`setAal2Cookie` at 4 route points) and **never renewed on the validated read path** (`getAal2UserId`). So a *continuously active* session was bounced to `/sign-in` ~1h after sign-in — an undocumented absolute cap masquerading as an inactivity window. The three "obvious" suspects (JWT/AAL1 expiry, refresh-token rotation, cookie persistence) were all ruled out by reading the actual code: AAL1 refresh + `sid` binding were correct; the bug was purely the AAL2 marker's missing renewal.
  - **Evidence:** `packages/passkey-2fa/src/aal2.ts:11` (`AAL2_TTL_SECONDS`, comment "re-challenge after"); `packages/passkey-2fa/src/guard.ts` (`getAal2UserId` had no renewal); `packages/passkey-2fa/src/routes.ts:264,293,333,355` (only mint points). Regression: `packages/passkey-2fa/src/aal2-renewal.test.ts`.
  - **Instance count in this log:** 1
  - **Recommended action:** When a guard mints a short-lived marker for a *long-lived* session, the renewal seam is part of the contract — pair "mint" with "slide on validated read" by default. Also: a TTL constant with an offhand "re-challenge after" comment is NOT an approved security policy; the absolute-timebox / idle-timeout decision belongs to Architect/PM. Flagged as an open Issue on the PR rather than baked in.

- **2026-06-24 · The PR #104 renewal fix was INCOMPLETE — renewal write swallowed in RSC renders → "forced logout" persisted; the regression test passed anyway (`polished-but-broken`)**
  - **Context:** `/fix` re-triage of the SAME forced-logout report after PR #104 shipped a sliding-renewal helper. Reproduced from the code: the bug was still live.
  - **Pattern surfaced:** PR #104 added `shouldRenewAal2Token` and called it from `getAal2UserId()`, but the renewal cookie WRITE there is wrapped in `try/catch {}` and Next.js **forbids `cookies().set()` during a Server Component render**. The browsing/read-only path is `(app)/layout.tsx → requireAal2() → getAal2UserId()` — a pure RSC render — so the renewal was **silently swallowed** and only ever persisted when the user hit a *mutating route handler* in the last 30 min of the hour. A user who only reads pages was still bounced at the 1h boundary. **Two compounding failures:** (1) the chosen renewal seam can't write on the path that matters; (2) the regression test (`aal2-renewal.test.ts`) tested the *pure helper* and *manually re-minted* the token in its own loop — it never exercised a real persistence seam, so it stayed green while the live defect persisted (the `mocked-auth-green` / `polished-but-broken` trap). The fix: a side-effect-free `renewedAal2Cookie` decision applied in the **middleware** (runs on every request incl. page navigations; CAN set response cookies), with a WebCrypto Edge twin (`renewedAal2CookieEdge`) so node:crypto stays out of the Edge bundle.
  - **Evidence:** `packages/passkey-2fa/src/guard.ts` (renewal write in `catch {}`, RSC-swallowed); `app/(app)/layout.tsx:11` (`requireAal2()` in a `force-dynamic` RSC render); old `packages/passkey-2fa/src/aal2-renewal.test.ts` (helper-only, self-re-minting loop — green while bug live). Fix + regression: `packages/passkey-2fa/src/{aal2-constants,aal2-edge,middleware}.ts`, `aal2-renewal-persist.test.ts`, `aal2-edge.test.ts`. Build artifact: `.next/server/middleware-manifest.json` (middleware `/` registered, matcher = all routes); `.next/server/middleware.js` (renewal present, 0 node:crypto).
  - **Instance count in this log:** 2 (same defect, two passes — the renewal-seam-vs-persistence-path mismatch)
  - **Recommended action (counter ≥2 — consider for codification at next role retro):** Name the anti-pattern **`unwritable-renewal-seam`** — a "renew on read" that lands on a path where the framework forbids the write (RSC `cookies().set()`) is a no-op masquerading as a fix. Two structural rules surfaced: (a) **a renewal/mutation seam must be chosen for where it CAN persist** (middleware / route handler / Server Action), never an RSC render; (b) **a regression test for a persistence bug must drive a real persistence seam, not a hand-re-minted in-loop simulation** — otherwise it's `mocked-auth-green` and will pass over the live defect. Also reinforces `[mechanical-output-verification]`: the Edge-bundle node:crypto failure was invisible to typecheck + 40 unit tests and only surfaced in `pnpm build` — the build is load-bearing, and the manifest/bundle inspection (renewal present + 0 node:crypto) is the real proof. **Flag for Automation (`[per-surface-vertical-test]`):** the auth→authz→render vertical (active-across-the-boundary on a browsing-only path = no logout; idle-past-window = re-challenge) belongs in E2E, exercised through real navigation + the middleware, not auth-mocked unit green.

- **2026-06-24 · "Forced logout" pass #3 — DISTINCT root cause: middleware double-reads the session (`getUser` + `getSession`) → doubled refresh-token rotation surface → Supabase revokes session under prefetch concurrency**
  - **Context:** `/fix` re-triage of the SAME forced-logout report after the renewal-seam fix (pass #2) shipped. The TTL/renewal axis was correct and well-tested, but the symptom recurred. Reproduced from the code by tracing the auth-call surface in the middleware, not from the user.
  - **Pattern surfaced:** The edge middleware read the Supabase session **twice per request** — `auth.getUser()` (the coarse gate + AAL1 refresh) **and then** `auth.getSession()` (only to recover `session_id` for AAL2 renewal). Each read is an independent **refresh-token-rotation trigger**. Supabase rotates the refresh token on use and **revokes the entire session on detected reuse** of an already-rotated token. The middleware runs on every request and Next.js fires **concurrent middleware invocations per navigation** (link prefetch + RSC/data request), so doubling the per-request rotation surface widened the race window where two in-flight requests present the same refresh token → spurious revocation → bounce to `/sign-in`. For a continuously-active single user this fires **stochastically — "every few hours"**, fully independent of the AAL2 TTL (which is why passes #1/#2 didn't resolve it). The three classic suspects re-ruled-out from code: AAL1 JWT expiry (refresh handled), cookie persistence (SSR cookies correct), `sid` rotation (Supabase `session_id` is stable across refresh — only the *call surface* was the problem). **Fix:** validate the session ONCE (`getSession()` returns both `user` for the gate AND `access_token` for `sid`), drop the second `getUser()` round-trip. The authoritative AAL1+AAL2 validation stays server-side in `requireAal2()`/`guard.ts getUser()` (Node runtime) — the edge gate was always intentionally coarse, so dropping its `getUser()` loses no security.
  - **Evidence:** old `packages/passkey-2fa/src/middleware.ts:73` (`getUser()`) + `:95` (`getSession()`) — two rotation triggers per request. Fix: `packages/passkey-2fa/src/middleware.ts` (single `getSession()`, `sid` derived from the same result). Regression: `packages/passkey-2fa/src/middleware-refresh.test.ts` (drives the real factory; asserts `getUser+getSession == 1` per request — failed `expected 2 to be 1` before the fix). Guard unchanged: `packages/passkey-2fa/src/guard.ts:21` still `getUser()` (authoritative path intact). Build artifact: `.next/server/middleware-manifest.json` (middleware `/` registered, all-routes matcher).
  - **Instance count in this log:** 3 (same reported symptom, three structurally-distinct root causes: absolute-TTL → unwritable-renewal-seam → doubled-rotation-surface)
  - **Recommended action (counter ≥3 — PROPOSE FOR CODIFICATION at next role retro):** Two things. (1) Name the anti-pattern **`redundant-rotation-trigger`** — calling more than one credential-rotating auth method per request (`getUser` + `getSession`, or repeated refreshes) multiplies the refresh-token-reuse-revocation surface; under framework prefetch concurrency this is a periodic forced-logout. Rule: **read/validate the session exactly once per request and derive everything (user, sid, claims) from that single result.** (2) Meta-pattern across all three passes: **a recurring user-facing symptom with multiple independent root causes needs the real auth→authz→render vertical under concurrency, or each fix only retires one cause and the symptom outlives it.** This is the strongest possible case for the `[per-surface-vertical-test]` flag below — three PRs is the cost of not having it. **Flag for Automation (`[per-surface-vertical-test]`):** the E2E must drive **concurrent/prefetched navigation** through the live middleware over a multi-refresh window (not a single sequential request) to catch the rotation-race the unit pin can only approximate.
