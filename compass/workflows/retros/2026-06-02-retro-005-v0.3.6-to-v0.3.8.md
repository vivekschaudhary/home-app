## Recurring anti-patterns (soft-spec-rationalization surfaces)

### `polished-but-broken` recurrence — the codification fell into its own trap

v0.3.6 codified `[mechanical-output-verification]` to close the gap between source intent and runtime artifact. The same codification anchored the Next.js check on `.next/server/middleware-manifest.json` — correct for Next 13–15, **wrong for Next 16+** where that file is empty by design.

**The pattern named the failure shape; the same failure shape immediately bit the pattern's own application.** Caught by external (crypto-app) review on Next 16 stack, not by internal Compass discipline.

This is **the recursive `polished-but-broken` failure mode** — framework patterns that close gaps inherit the same risk at the codification layer. The fix is the same as the original: mechanical inspection at the framework-output layer. Worth examining whether codification entries themselves should carry framework-version anchors going forward.

### Scope-creep on deferred roadmap — caught by user, not framework

v0.3.8 was originally planned with L2 = 8 per-agent adapter docs in a new `compass/agents/` directory. **That's scope creep** — Compass would have duplicated LiteLLM's documentation. User caught it within an hour: *"we are not creating per agent adapter we will use an existing adapter like litellm and other competitors for the adapters."*

Same-day correction shipped within an hour of the catch. **Framework discipline + user judgment compose; framework alone scope-creeped.**

Worth naming explicitly: **the framework's load-bearing oversight is the user.** Future v0.4+ multi-agent architecture (orchestrator vision) would need to preserve this user-as-oversight role explicitly, not replace it with orchestrator agency.

## Real-world artifact analysis (NEW — per user request)

This section sets precedent: future retros can include in-cycle artifact criticism when the user surfaces specific artifacts or when artifacts become structurally load-bearing for framework outcomes. Below: honest 1–10 ratings + 3 best + 3 overkill (where requested) + recommendations.

### `compass/roles/reviewer.md` — **rating 7/10**

**Why 7/10:** The file does the right things structurally — clear contract with the Codex agent (the "Expected Codex output shape" section), freshness markers (`last_verified`, `external_source`), named anti-patterns with v0.3.6 additions (`polished-but-broken`, `direct-import-test-suspicious`, `narrow-bug-focus`), and explicit Step 0 framework-registration discipline. **Produces real value when Codex executes against it** (CB-1.4 caught the CVE-2025-29927 + defense-in-depth gap; crypto-app review caught Next 16 anchor error in v0.3.6 codification). The strengths are load-bearing.

But the file grew significantly in v0.3.6 — Step 0 is dense, Step 4 is operationally heavy, the anti-patterns section went from 5 to 10 items. **Reading + cognitive cost on every Codex review invocation is rising.** Risk: "too many named patterns become noise rather than signal" (the failure mode Principle #14 itself warns against when applied to its own structural enforcement).

#### Three things that work best

1. **Step 0 framework-registration check is concrete and mechanical.** Names specific manifest files (correctly cited Next 16 anchor after PR #1: `.next/server/functions-config-manifest.json`) with verifiable criteria ("confirm the `/_middleware` entry exists with `runtime: \"nodejs\"` + matchers"). The REQUIRED-vs-OPTIONAL gating ("REQUIRED for routing-layer / discovery-layer / framework-convention changes; OPTIONAL for pure logic changes") prevents over-application. **Exactly the shape Principle #14 calls for** — explicit imperative + named failure mode + verification mechanism.

2. **"Expected Codex output shape" section is the freshness target.** Explicit contract that workflows can parse against. Field-by-field expectations make drift detectable. The `external_source: https://github.com/openai/codex` frontmatter ties freshness to a real-world signal. **First reviewer-role artifact in Compass with structural review-time invariants** — and v0.3.7's freshness detection ran productively against it.

3. **Named anti-patterns inherit vocabulary across reviews.** `polished-but-broken`, `direct-import-test-suspicious`, `narrow-bug-focus` give Codex (and human readers) memorable handles for failure modes. The fact that `narrow-bug-focus` was **self-named by Codex during CB-1.4 retrospective** is particularly strong — the role file teaches the agent its own blind spots, then the agent contributed back to the framework's vocabulary.

#### Three things that are overkill

1. **Step 0 is too front-loaded for general PRs.** Lists 4 framework-specific anchors (Next.js / Vercel Functions / Expo / general principle) + cross-reference to `direct-import-test-suspicious` + REQUIRED-vs-OPTIONAL determination logic. For a pure-logic bug fix that doesn't touch framework-discovered surfaces, Codex must read all of this before concluding "OPTIONAL." That's wasted cognitive budget on every review. **Could conditionalize** with a quick decision tree at the top of Step 0: "Does this PR touch framework-discovered surfaces (file-based routing, middleware auto-registration, plugin discovery, asset bundling)? If NO → skip Step 0; proceed to Step 1. If YES → continue Step 0."

2. **Step 4 review-time freshness is operationally expensive.** Codex must "re-verify against current primary docs" every load-bearing framework claim. In practice that means hitting Anthropic / OpenAI / Google / Vercel docs for every review with framework-behavior claims. **Cost + latency rise.** Worse: under pressure (large PR, time constraint), this is the exact step Codex is most likely to silently skip — which is the soft-spec-rationalization failure mode Principle #14 warns against. **Could scope tighter:** re-verify CLAIMS that are NEW in this PR's story (i.e., claims not already verified in prior PRs against the same external source), not every load-bearing claim ever made.

3. **Anti-patterns section is 10 items now, growing.** Multiple anti-patterns reference v0.3.5 / v0.3.6 / canon entries — provenance is rich but cumulative cognitive load is significant. **Consolidation candidates:** `polished-but-broken` already encompasses `direct-import-test-suspicious` (test passes while framework doesn't register the file) and `narrow-bug-focus` (focusing on functional vs framework-registration is the same gap). Could fold these into `polished-but-broken` as examples rather than separate named anti-patterns. Net: 5–6 anti-patterns instead of 10.

#### Recommendations

- **v0.3.9 or v0.3.10 cleanup pass:** trim Step 0 with a decision tree; scope Step 4 tighter (NEW-claims-only); consolidate 3–4 of the redundant anti-patterns under `polished-but-broken` as examples.
- **Honest framing:** reviewer.md is succeeding in producing value; the pruning is about reducing per-review cognitive cost, not about removing functionality.
- **Watch:** if reviewer.md grows further in v0.3.9 / v0.3.10 without pruning, the size-vs-signal warning compounds. Add a soft cap (e.g., "if anti-patterns section exceeds 10 entries, prune by consolidation before adding more") as a soft convention.

### CB-1.5 story (sign-out endpoint, `POST /api/auth/sign-out`) — **rating 9/10**

**Why 9/10:** Unusually thorough story work. The structure matches Compass discipline at every level: architecture references with section anchors, brief cross-references, library API contracts with annotations, explicit "why this isn't public" rationale, comprehensive DRI Log (4 decisions with alternatives + reversibility + 3 risks with likelihood/impact/mitigation), and **explicit application of v0.3.6 `[mechanical-output-verification]` with the corrected Next 16 anchor (AC 11 cites `app-paths-manifest.json`, NOT legacy `pages-manifest.json`)**. It's the kind of story Codex review will close cleanly because the framework discipline is built in at story-write time, not retrofit at review time.

#### Particularly strong elements

- **AC 11 correctly applies v0.3.6 + PR #1 in the wild.** First in-the-wild evidence that the consumer-side application of v0.3.6 codification + PR #1's Next 16 anchor correction is happening correctly. **Concrete framework→consumer round-trip** — the framework's correction propagated to the consumer's story-writing within hours.
- **DRI Decision rationale is real.** Each of 4 decisions includes alternatives considered + reversibility. Decision #2 ("behind the proxy gate") is especially load-bearing and the rationale traces directly to architecture invariants ("cookie alone is not trusted — DB row is the source of truth"). Decision #1 ("sign-out scope is 'this session only'") closes the brief Open Issue with explicit deferral reasoning.
- **AC 7 "cookie-attribute parity test"** — names the constant-duplication risk explicitly and ships a mechanical guard ("programmatically parse the `Set-Cookie` header from both the issuance and the sign-out responses and assert the attribute set is identical except for `value` and `Max-Age`"). The kind of test that prevents a class of future bugs, not just this one.
- **"Why sign-out is BEHIND the proxy gate" Tech notes section** — preempts the intuitive misread before any reviewer flags it. Three reasons named + load-bearing source-code marker convention. **Direct application of `[role-boundary]` source-code marker pattern (v0.3.4).** The "future readers (including future-me) can't re-derive the wrong answer" framing in DRI Decision #2 is honest about cognitive load and explicit about countermeasure.
- **AC 12 "No library extension this round"** — explicit guidance on when extraction IS in-scope ("if it strengthens AC 7's cookie-attribute parity check by making attribute drift mechanically impossible"). Engineer judgment scope made operational.
- **Out-of-scope is comprehensive** — 6 items each with deferred-to anchor (CB-1.6, post-MVP, runbook). Prevents engineer-time scope creep.
- **Race-condition risk (Risk #2)** acknowledged at honest likelihood/impact levels — explicit "Postgres serializes per-row writes" reasoning, with the AC 7 idempotency test as the safety net. Doesn't over-engineer (transaction wrapper) for an n=1 concurrency floor.

#### Three improvement recommendations

1. **Consolidate the test-related ACs.** AC 7 (Vitest), AC 8 (Codex E2E), AC 9 (typecheck), AC 10 (lint), AC 11 (production build + artifact inspection) overlap conceptually — they're all "CI verification gates" with different mechanics. **Could collapse to one composite "CI gate" AC with sub-bullets**, leaving AC 7 + AC 8 as the substantive test ACs that actually exercise behavior. Net: ~10 ACs instead of 12; story scans faster without losing rigor. The cookie-attribute parity test (currently AC 7) and the E2E sign-out → dashboard-redirect → re-sign-out flow (currently AC 8) are the load-bearing tests; the others are gates.

2. **AC 12 isn't really an AC — it's a Decision.** "No library extension this round" with optional in-scope extraction guidance is engineering judgment scope, not a verifiable acceptance criterion. **Move to DRI Decisions** as "Engineer extension judgment: in-scope iff X; otherwise inline." That keeps the AC list mechanical (each AC verifiable at code-review time) and the DRI Log carries the scope-discipline reasoning.

3. **The story is long (~400+ lines).** Some DRI Log Decision rationale (Decisions #2 and #4 especially) could move to PR descriptions or commit messages rather than living in the story file. **The architecture/brief references at the top + the "Why behind-the-proxy" Tech notes already carry the core why;** full alternatives-considered detail can live where it's encountered (PR review). Net: ~250-line story stays scannable; PRs carry the bet-context rationale. The DRI Log Risks (3 items) should stay in the story since they're forward-looking, not historical.

#### Net read

High-quality story work. The thoroughness is **appropriate** for an auth endpoint (high stakes, multiple failure modes, cross-bet dependency on CB-1.4's proxy enforcement). The DRI logging + cross-bet linking (CB-1.4 Engineer DRI Decision #5 explicitly cited) + framework application (v0.3.6 + PR #1 correction in AC 11) are exactly what Compass discipline calls for. **The recommendations above are about scannability for future readers, not about correctness — the story is structurally correct.** Codex review on this PR will be cleaner because of the upfront work.
