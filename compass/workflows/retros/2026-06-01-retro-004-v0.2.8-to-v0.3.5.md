## Patterns surfaced (positive)

### One-Compass-original-per-session cadence held across 5 sessions

v0.3.1 → v0.3.5. Each session shipped exactly one substantive Compass-original (or structural section). No scope creep. **The deliberate-pace discipline survived contact with momentum.** Worth noting: v0.3.5 added a same-day Medium-scope extension based on CB-1.4 feedback — that's the first time cadence allowed in-session extensions; the precedent should hold (same-day extensions are valid when grounded in real-world friction from the same session's substantive work, but should not become routine).

### Five-shape Compass-originals catalog stabilized

Catalog now spans:

| Shape | Members | What it answers |
|---|---|---|
| Enforcement | cite-or-mark-n/a · refuse-escalate · soft-spec-hardening | What the workflow REQUIRES |
| Interaction | elicitation-with-options | How the workflow ASKS |
| Freshness | freshness-check | How the workflow STAYS CURRENT |
| Observability | role-boundary | How the workflow EXPOSES STRUCTURE |
| Handoff | agent-handoff | How the workflow ROUTES across agents |

**Catalog balance: 3 enforcement vs 4 usability** (interaction + freshness + observability + handoff). The split between "what the framework demands" vs "how the framework makes itself usable" is becoming the load-bearing organization axis. Worth validating against the next 2-3 Compass-originals to confirm the framing holds.

### Codex's own retrospective (CB-1.4) — 5 self-improvement points

User ran CB-1.4 through both reviewers and reported Codex's verbatim self-critique. Preserved here for future reference:

> 1. Start with framework registration checks before reading functional tests. For Next proxy/middleware work, verify file placement, build output, and generated manifest first.
> 2. Treat direct-import tests as suspicious when the feature depends on framework discovery. Here, `import proxy from "@/app/proxy"` let the suite pass even if Next never registers the file.
> 3. Re-verify any "load-bearing" framework claim from the story against current primary docs, especially when it names runtime behavior or file conventions.
> 4. Check ACs for internal contradictions earlier. In this PR, AC 1, AC 7, and the DRI decisions conflicted on request-vs-response headers and on whether handlers should re-verify.
> 5. Prefer "is this actually deployed by the framework?" over "do the tests pass?" for routing-layer changes.
>
> Net: my review found a real bug, but it was too narrow. The other agent reviewed at the correct abstraction level first: framework registration, runtime legality, then request semantics. That is the main improvement Codex should adopt.

**These are gold.** Each point is a candidate for direct integration into `compass/roles/reviewer.md` and/or new Compass-originals — see Convention candidates below.

## Recurring anti-patterns (soft-spec-rationalization surfaces)

### `polished-but-broken` — formally named in v0.3.5

Tests pass + build succeeds + narrative coherent + principles cited + behavior wrong. Subtler form of #14's rationalization surface than prior named instances (silent summarization, log-and-walk-away, smart defaults). The agent passes increasingly sophisticated structural checks while the actual artifact diverges from intent. **Pattern recognition:** the rationalization surface gets subtler as the framework hardens — there is always a soft spec to be found. New surfaces will emerge.

### Roadmap commitment slips (freshness detection)

v0.3.3 committed freshness detection to v0.3.4. v0.3.4 bumped to v0.3.5+. v0.3.5 bumped to v0.3.6+. **3 consecutive slips.** Each slip was individually defensible (token tracking and handoff automation were higher-leverage given user friction). But the pattern is itself a rationalization shape: the next-release commitment is a soft spec; the substantive next session re-prioritizes; the original commitment defers. **This is the same shape as the retro overdue meta-finding** — soft commitments to future-self get rationalized away by present-self.

### Retro cadence rationalization (this retro is the example)

Retro promised after improvement #20. Fired at #22. Two consecutive opportunities passed (v0.3.4 release, v0.3.5 release). The deferral logic: "the next substantive work is more important than reflection." **Same shape as above.** Counter-correction structurally: the next retro counter is bumped explicitly + this slip is named.
